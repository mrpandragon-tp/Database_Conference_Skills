# Section Blueprint

## Abstract

- problem tension
- gap in current methods
- proposed artifact and core mechanism
- key quantified evidence
- boundary-aware takeaway

Every Abstract claim must map to a main-paper figure, table, or theorem.

## Introduction

Use paragraph roles rather than a literature dump:

1. context, workload, and system requirement
2. concrete demanding instance
3. prior approaches and unresolved problem, with the failure in the topic sentence
4. technical opportunity and why it is insufficient by itself
5. explicit proposal sentence defining the method and core idea
6. first method idea
7. second method idea, correctness, update, or integration
8. evidence summary, contribution bullets, and roadmap

Checks:

- one principal message per paragraph
- topic sentence summarizes paragraph
- containment and input relationships are explicit
- first method mention defines name, abbreviation, artifact type, and target
- teaser figure is close to the proposal

## Method

- formal setting
- complete build/query/update/verification path
- mechanism with pseudocode, formulas, examples, and figures as needed
- correctness and complexity obligations near the corresponding mechanism
- implementation details sufficient to establish a complete database artifact

## Experiments

- research questions and claim mapping
- setup and fairness
- joint quality/query/build/update/memory results as claimed
- ablations that separate core innovation from systems engineering
- sensitivity and boundary cases
- limitations

## Related Work and Conclusion

- organize Related Work by the paper's problem and claim-prior delta
- cite each named method or factual claim locally
- make Conclusion restate supported contributions and boundaries, not introduce a broader unsupported lesson
